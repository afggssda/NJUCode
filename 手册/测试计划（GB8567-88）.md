## 1 引言

### 1.1 编写目的

本文档依据 GB8567-88 软件文档编制规范中“测试计划”的常见结构编写，用于说明 NJUCode 项目的测试目标、测试范围、测试方法、测试环境、测试项目、测试用例设计、通过准则和风险控制。

本文档面向以下人员：

- 项目开发人员：用于明确需要验证的功能点和质量要求。
- 测试人员：用于执行测试、记录结果和跟踪缺陷。
- 课程评审人员：用于了解项目测试设计的完整性。
- 项目维护人员：用于后续版本迭代时开展回归测试。

### 1.2 背景

NJUCode 是一个基于 Textual 的终端智能代码助手，主要支持本地代码分析、技能命令、上下文压缩、补丁应用与回滚、项目健康诊断、任务扫描、代码复杂度度量等功能。

当前项目已实现以下与测试直接相关的能力：

- `/doctor`：项目健康诊断命令。
- `test_all_features.py`：覆盖主要功能的 unittest 全量回归测试脚本。
- 测试报告自动生成：运行 `python test_all_features.py` 后生成 Markdown 和 JSON 报告。
- 覆盖率统计：运行 `python test_all_features.py --coverage` 后生成覆盖率 Markdown、JSON 和 HTML 报告。

### 2.3 不在本轮测试范围内的内容

| 内容 | 原因 |
| --- | --- |
| 真实大模型 API 调用 | 会依赖网络、API Key、余额、限流和模型服务状态，不利于离线可复现测试 |
| 真实 MCP 外部服务连接 | 外部命令和服务环境不稳定，本轮只测试配置和适配逻辑 |
| Textual UI 自动点击测试 | 当前测试以本地服务和命令逻辑为主，未接入 UI 自动化框架 |
| 性能压力测试 | 当前版本以功能正确性和工程稳定性为主要验收目标 |


### 2.4 测试策略

本项目采用以下测试策略：

| 策略 | 说明 |
| --- | --- |
| 离线优先 | 测试不依赖真实网络请求，保证课堂环境和本地环境可重复执行 |
| 单元测试与集成测试结合 | `test_all_features.py` 既覆盖单个模块行为，也覆盖命令和技能入口 |
| 真实项目文件检查 | `/doctor` 直接检查当前项目文件，发现项目结构和文档漂移 |
| 临时工作区隔离 | patch、settings、metrics、tasks 等测试使用临时目录，避免污染真实项目 |
| 自动报告生成 | 每次运行全量测试后自动保存 Markdown 和 JSON 报告；启用 coverage 时额外保存覆盖率 Markdown、JSON 和 HTML 报告 |
| 安全优先 | 测试覆盖 API Key 不落盘、路径穿越拒绝、陈旧 patch 拒绝等安全场景 |

## 3 测试设计说明

| 8 | `context_compressor` | 检查上下文压缩和 token 估算 |
| 9 | `settings_store` | 检查设置保存、备份、session 导入导出 |
| 10 | `patch_engine` | 检查 patch 生成、应用、回滚和路径安全 |
| 11 | `skills` | 检查内置技能注册、命令别名和 `/help` 输出 |

## 5 评价准则

### 5.1 进入测试准则

满足以下条件后可以进入测试：

- 项目代码可以正常导入。
- `requirements.txt` 存在。
- Conda 环境 `projlab` 可用。
- `test_all_features.py` 存在。
- 项目根目录为 `D:\nju\NJUCode`。

### 5.2 测试通过准则

| 项目 | 通过标准 |
| --- | --- |
| 全量测试 | `python test_all_features.py` 运行结果为 OK |
| 失败用例 | fail 和 error 数量为 0 |
| 跳过用例 | 只允许因环境限制产生的合理 skip |
| 报告生成 | `.nju_code/reports/` 下生成 Markdown 和 JSON 测试报告 |
| 覆盖率报告 | 启用 `--coverage` 时生成覆盖率 Markdown、JSON 和 HTML 报告；核心回归范围语句覆盖率不低于 90%，分支覆盖率不低于 60% |
| Doctor | 核心检查项不出现 fail |
| 安全测试 | API Key 不应持久化，路径穿越 patch 应被拒绝 |

### 5.3 暂停测试准则

出现以下情况时暂停测试：

- 项目无法导入或测试脚本无法启动。
- 测试环境缺少必要依赖。
- 大量用例因同一基础问题失败。
- 测试过程破坏项目真实配置或真实文件。

### 5.4 恢复测试准则

满足以下条件后恢复测试：

- 修复导致测试无法运行的基础问题。
- 清理异常临时文件。
- 确认测试仍在临时目录中执行破坏性操作。
- 重新运行全量测试并生成新报告。

## 6 风险与控制

| 风险 | 影响 | 控制措施 |
| --- | --- | --- |
| 测试依赖真实网络 | 可能导致测试不稳定 | 当前不进行真实大模型 API 调用 |
| API Key 泄露 | 安全风险 | 测试 API Key 不落盘，doctor 检查持久化密钥 |
| patch 测试误改真实文件 | 破坏项目 | 使用临时目录和路径穿越校验 |
| Windows 沙箱限制删除文件 | 个别删除/清理用例跳过 | 将此类 skip 视为环境限制，不计为功能失败 |
| README 与命令不同步 | 展示时造成误解 | 文档一致性测试和 doctor README 检查 |
| 新功能未纳入测试 | 回归风险 | 新增功能需补充 unittest 和 doctor 检查 |
