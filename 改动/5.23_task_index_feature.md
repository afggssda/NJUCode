# 2026-05-23 Task Index 任务扫描功能改动记录

本文档记录 2026-05-23 对 `NJUCode` 项目新增的非测试类功能：项目任务/TODO 扫描器。该功能用于从项目源码和 Markdown 文档中集中发现待办事项、风险标记和清单项，方便开发者快速了解当前工程中还没有完成的工作。

## 一、改动目标

本次新增功能不属于软件测试能力，而是一个面向日常开发效率的项目分析功能。

核心目标：

1. 扫描项目中的 `TODO`、`FIXME`、`BUG`、`HACK`、`NOTE` 标记。
2. 扫描 Markdown 中的任务清单项，例如 `- [ ]` 和 `- [x]`。
3. 将扫描结果接入聊天斜杠命令、内置 Skill 和 Tools 面板。
4. 支持按标签、负责人、完成状态和数量上限过滤结果。
5. 为新增功能补充综合测试，确保服务层、命令层和 Skill 层都能正常工作。

## 二、新增文件

### 2.1 `frontend/services/task_index.py`

新增 `ProjectTaskIndex` 服务，用于构建项目任务索引。

主要能力：

- 扫描文本类文件中的任务标记。
- 支持标记格式：
  - `TODO: message`
  - `TODO(owner): message`
  - `FIXME: message`
  - `BUG(owner): message`
  - `HACK: message`
  - `NOTE: message`
- 支持 Markdown 清单：
  - `- [ ] unfinished task`
  - `- [x] finished task`
- 输出结构化 payload，类型为 `task_index`。
- 根据标签自动设置优先级：
  - `BUG`、`FIXME` 为 high。
  - `TODO`、`HACK`、`CHECKBOX` 为 medium。
  - `NOTE` 为 low。
- 默认排除 `.git`、`.nju_code`、`.test_tmp`、`.venv`、`node_modules`、`__pycache__` 等目录，避免扫描缓存、临时文件和运行状态文件。

主要数据结构：

```python
@dataclass
class TaskItem:
    path: str
    line: int
    tag: str
    text: str
    status: str
    priority: str
    source: str
    owner: str = ""
```

主要入口：

```python
ProjectTaskIndex(workspace_root).scan(
    tag="",
    include_done=False,
    owner="",
    limit=50,
)
```

返回内容包含：

- `summary`: 总数、open 数、done 数、扫描文件数、按 tag/status/priority 聚合的统计。
- `filters`: 本次扫描使用的过滤条件。
- `items`: 任务列表，每项包含文件路径、行号、标签、文本、状态、优先级和负责人。
- `total_before_limit`: 应用数量限制前的结果总数。

## 三、修改文件

### 3.1 `frontend/services/code_analysis.py`

新增 `task_index` payload 的文本格式化逻辑，使任务扫描结果可以在聊天窗口中直接阅读。

新增斜杠命令：

```text
/tasks [--tag TODO|FIXME|BUG|HACK|NOTE|CHECKBOX] [--owner name] [--done] [--top 50]
```

支持参数：

- `--tag`: 只显示指定标签。
- `--owner`: 只显示指定负责人，例如 `TODO(alice)`。
- `--done` / `--all` / `--include-done`: 包含已完成的 Markdown 清单项。
- `--top` / `--limit`: 限制返回数量，默认 50，最大 200。

### 3.2 `frontend/skills/builtin/__init__.py`

Skill 信息：

- `category`: `analysis`
- `permissions`: `READ_ONLY`

def execute_tasks(analyzer: Any, params: Dict[str, Any]) -> Dict[str, Any]:

## 五、当前已知问题与后续优化方向

第一次运行 `/tasks` 后发现扫描规则仍偏宽松，会把文档、测试样例、字符串常量和正则表达式中的 `TODO/FIXME/BUG` 也识别为任务。

典型误报包括：

- README 中的命令参数枚举。
- `task_index.py` 中扫描器自身的正则表达式。
- `test_all_features.py` 中用于测试扫描器的样例任务。
- 普通英文注释中出现的 `bug` 单词。

后续建议优化：

1. Python 文件优先基于 `tokenize` 只扫描注释，不扫描字符串和代码正文。
2. 要求代码任务标记必须出现在注释前缀之后，例如 `# TODO:`、`# FIXME:`。
3. Markdown 默认只扫描 checkbox，或只扫描独立行里的 `TODO:` / `FIXME:`。
4. 默认排除 `test_*.py` 中的样例任务，并提供 `--include-tests` 选项。
5. 避免识别命令说明中的 `TODO|FIXME|BUG|...` 枚举。

## 六、本次功能价值

该功能使 `NJUCode` 除了代码检索、影响面分析、Project Doctor 自检之外，又具备了一个面向项目维护的开发辅助能力。用户可以直接在聊天框或 Tools 面板中查看当前项目待办事项，快速定位未完成工作、风险标记和 Markdown 清单任务。

## 七、2026-05-23 追加优化：降低 `/tasks` 误报

根据首次运行 `/tasks` 看到的结果，本次继续收紧任务扫描规则，重点解决 README 命令说明、正则表达式、字符串常量、普通英文 `bug` 单词和测试样例任务造成的误报。

### 7.1 扫描规则优化

`frontend/services/task_index.py` 中新增更严格的解析策略：

1. Python 文件改为使用标准库 `tokenize`，只扫描 `COMMENT` token，不再扫描字符串、正则、代码正文或 docstring。
2. `TODO/FIXME/BUG/HACK/NOTE` 必须使用明确任务格式：

```text
TODO: message
TODO(owner): message
FIXME: message
BUG(owner): message
```

3. 不再识别普通文本中的裸词，例如普通英文句子里的 `bug` 不会被当作任务。
4. Markdown 文件继续支持：

```text
- [ ] unfinished task
- [x] finished task
TODO: markdown task
```

5. 默认排除测试文件和测试目录：

```text
test_*.py
*_test.py
tests/
```

如确实需要扫描测试文件，可使用：

```text
/tasks --include-tests
```

### 7.2 命令和 Skill 参数更新

`/tasks` 和 `/todos` 新增参数：

```text
--include-tests
```

更新后的命令格式：

```text
/tasks [--tag TODO|FIXME|BUG|HACK|NOTE|CHECKBOX] [--owner name] [--done] [--include-tests] [--top 50]
```

`builtin.tasks` Skill 也新增 `include_tests` 参数，用于在 Skill 层控制是否扫描测试文件。

### 7.3 测试补充

`test_all_features.py` 中的 `ProjectTaskIndexTests` 增加了误报防护断言：

- Python 字符串里的 `TODO: ...` 不应被扫描。
- 普通注释中出现英文 `bug` 不应被扫描。
- README 风格的 `TODO|FIXME|BUG|...` 枚举不应被扫描。
- `test_noise.py` 默认不应被扫描。
- 显式传入 `include_tests=True` 后，测试文件任务才会出现。
- 文本输出中显示 `include_tests=False`，方便用户理解当前过滤状态。
