# 2026-05-22 Project Doctor 与综合测试改动记录

## 二、新增文件

### 2.1 `frontend/services/project_testing.py`

新增 Project Doctor 服务层，用于执行项目级自检和生成结构化报告。

主要数据结构：

- `ProjectIssue`：单条诊断问题，包含严重级别、消息、路径和修复提示。
- `ProjectCheckResult`：单项检查结果，包含状态、摘要、耗时、指标和问题列表。
- `ProjectTestReport`：完整项目自检报告，包含通过数、警告数、失败数、跳过数，以及 JSON/Markdown 输出。
- `ProjectTestRunner`：项目自检执行器。

主要检查项：

| 检查项 | 说明 |
|------|------|
| `layout` | 检查项目关键目录和文件是否存在 |
| `requirements` | 检查依赖声明是否完整 |
| `python_syntax` | 使用 AST 解析所有 Python 文件 |
| `imports` | 检查 `requirements.txt` 中依赖是否可导入 |
| `entrypoint` | 检查 `main.py` 和 `NjuCodeApp` 入口可导入 |
| `code_analysis` | 覆盖 scan/search/symbol/summary/deps/recall/impact |
| `code_extractor` | 检查 AI 回复代码块提取和多文件拆分 |
| `context_compressor` | 检查 token 估算和上下文压缩 |
| `settings_store` | 检查 settings 保存、备份、会话导入导出 |
| `patch_engine` | 检查 patch 生成、预览、应用、回滚、路径安全 |
| `skills` | 检查内置技能注册、命令别名唯一性和 help 输出 |
| `mcp` | 检查默认 MCP server presets |
| `ui_structure` | 检查核心 Tab、快捷键、Doctor 按钮 |
| `readme_commands` | 检查 README 是否包含新增命令 |
| `security` | 检查 API key 是否被持久化、是否存在本地 `.env` |
| `line_budget` | 统计项目代码行数和主要文件规模 |

对外函数：

- `run_project_test_suite(...)`
- `report_to_text(...)`
- `run_doctor_as_payload(...)`

这些函数用于被 Skills、斜杠命令、测试文件或未来 CLI 入口复用。

### 2.2 `test_all_features.py`

新增综合测试文件，使用 Python 标准库 `unittest`，不依赖 pytest。

覆盖范围：

- 仓库结构与关键文件存在性
- 所有 Python 文件 AST 语法解析
- `requirements.txt` 依赖声明检查
- `main.py` / `NjuCodeApp` 入口导入
- `CodeAnalyzer` 的扫描、搜索、符号检索、摘要、依赖图、召回、影响面分析
- `code_extractor` 的代码块提取、文件名识别、多文件拆分、shell block 过滤
- `ContextCompressor` 的 token 估算、压缩、压缩历史统计
- `SettingsStore` 的保存、备份、导入导出和导出清理
- `PatchEngine` 的 diff、语法校验、序列化、历史持久化、应用、回滚、路径穿越拦截
- Skills 注册、参数校验、权限检查和内置技能执行
- `AppState` 会话生命周期、API key 不落盘、Patch wrapper、Agent Skill 上下文选择
- MCP 默认 server presets、MCP tool toggle、MCP tool adapter
- OpenAI 兼容客户端空 API key 行为和文件上下文注入
- Project Doctor 自身的报告生成、命令调用、内置技能调用
- README 命令和运行方式检查
- `hello_world.py` 与 runtime tool 检查

测试运行命令：

### 3.4 `frontend/services/patch_engine.py`

增强 Patch 历史持久化兼容性。

原逻辑使用：

```python
tmp_path.replace(self.history_path)
```

在部分 Windows 沙箱环境中，即使普通写入允许，`replace()` 也可能被拒绝。现新增 fallback：

1. 优先使用临时文件 + replace 的原子写入。
2. 如果 `OSError`，回退为直接写入 `patch_history.json`。
3. 尝试清理临时文件。

这样可以保证在受限课程实验环境里 Patch 历史仍能保存。

## 六、注意事项

1. 当前测试文件会在仓库内临时创建 `.test_tmp`，Project Doctor 会临时创建 `.nju_code/test_tmp`，测试结束后会尽量清理。
2. 当前运行环境限制了文件删除操作，因此涉及 `unlink` 的两个测试会自动 skip。
3. Project Doctor 的 `security` 检查不会读取或输出 API key，只会提示本地 `.env` 文件存在。
4. `/test-all` 当前与 `/doctor` 走同一套项目自检服务；`test_all_features.py` 则是更完整的终端回归测试文件。

## 2026-05-24 测试报告生成增强

`python test_all_features.py` 现在不再只输出终端结果，测试结束后会自动在 `.nju_code/reports/` 下保存两份报告：

```text
.nju_code/reports/test_all_features_*.md
.nju_code/reports/test_all_features_*.json
```

报告内容包括：

- 测试总数、通过数、失败数、错误数、跳过数
- 开始时间、结束时间和耗时
- 非通过测试的原因
- 最慢的 10 个测试
- 每个测试用例的状态和耗时
