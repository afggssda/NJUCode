# 2026-05-23 Code Metrics 代码指标分析功能改动记录

本文档记录 2026-05-23 新增的高阶静态分析功能：代码复杂度与维护热点分析。该功能通过 AST、依赖图和环检测为项目提供更有技术含量的维护指标，而不只是简单的文本搜索或文件统计。

## 一、功能目标

新增 `/metrics` 命令，用于回答以下问题：

1. 哪些 Python 文件复杂度最高，最值得优先重构。
2. 哪些函数/方法拥有较高圈复杂度。
3. 哪些文件在内部依赖图中 fan-in / fan-out 较高。
4. 项目是否存在 import 环依赖。
5. 是否需要把测试文件纳入分析。

## 二、新增文件

### `frontend/services/code_metrics.py`

新增 `ProjectMetricsAnalyzer` 服务，主要技术点如下：

- 使用 `ast.parse` 解析 Python 文件。
- 使用自定义 `ComplexityCounter` 统计函数圈复杂度。
- 使用 `FunctionCollector` 收集函数、方法、限定名、行号和复杂度。
- 使用 `tokenize` 统计代码行、注释行和空行。
- 构建项目内部 import 依赖图。
- 支持相对 import 的基础解析。
- 使用 Tarjan strongly connected components 算法检测 import 环。
- 根据复杂度、代码行数、fan-in、fan-out 和环依赖计算 `hotspot_score`。

输出 payload 类型：

```text
code_metrics
```

主要输出字段：

- `summary`: 文件数、代码行数、类数量、函数数量、依赖边数量、环数量、复杂函数数量。
- `hotspots`: 按维护风险排序的文件列表。
- `complex_functions`: 按复杂度排序的函数/方法列表。
- `cycles`: import 环依赖列表。

## 三、命令接入

### `frontend/services/code_analysis.py`

新增命令：

```text
/metrics [--top 10] [--include-tests]
```

参数说明：

- `--top`: 控制热点文件、复杂函数和环依赖的返回数量。
- `--include-tests`: 将 `test_*.py`、`*_test.py` 和 `tests/` 纳入分析。

新增 `to_text()` 格式化逻辑，聊天窗口会展示：

- 项目总体指标。
- 热点文件排序。
- 最复杂函数列表。
- import 环依赖列表。

## 七、功能价值

相比普通代码搜索，`/metrics` 能从结构层面分析项目维护风险，体现了更强的工程技术含量：

- AST 静态分析。
- 圈复杂度计算。
- 内部依赖图构建。
- Tarjan 环依赖检测。
- 多指标热点评分。

该功能适合在课程展示中说明项目不仅能调用模型和检索文本，还具备本地代码理解、结构分析和维护风险定位能力。
